## 3. 指标口径基线

以下口径是当前可执行基线，不等于永久不可变，但在进入页面开发前需要先锁定。

| 指标 | 执行口径 | 备注 |
|---|---|---|
| 总营收 | 平台标准收入字段求和 | 美团=`营业收入`；饿了么=`收入`；京东=`收入` |
| 总订单量 | `有效订单` 求和 | 三平台均可取 |
| 活跃门店数 | 统计周期内 `有效订单 > 0` 的去重门店数 | 若后续要改成 `收入 > 0`，必须统一全平台一起改 |
| 覆盖省份数 | 统计周期内去重省份数 | 京东需补城市到省份映射 |
| 覆盖城市数 | 统计周期内去重城市数 | 需先做城市名标准化 |
| 平台营收贡献占比 | 平台总营收 / 三平台总营收 | 用于饼图 |
| 平台订单量占比 | 平台总订单量 / 三平台总订单量 | 用于对比表 |
| 客单价 | `顾客实付` / `有效订单` | 饿了么原字段为 `顾客实付总额` |
| 曝光量 | `曝光人数` 求和 | 不用 `曝光次数` 作为主口径 |
| 到手率 | `收入 / 营业额` | 三平台统一按该公式执行 |
| 门店转化率 | `下单转化率` | 门店转化率 Top20 按此字段执行 |

## 5. 字段标准化基线

### 5.1 建议统一后的核心字段

| 标准字段 | 美团 | 饿了么 | 京东 |
|---|---|---|---|
| platform | 固定值 `美团` | 固定值 `饿了么` | 固定值 `京东` |
| biz_date | `日期` | `日期` | `日期` |
| platform_store_id | `门店id` | `门店编号` | `门店id` |
| platform_store_name | `门店名称` | `门店名称` | `门店名称` |
| province | `省份` | `省份` | 需映射补齐 |
| city | `门店所在城市` | `城市名称` | `城市` |
| district | `区县市` | `区县名称` | 暂缺 |
| revenue | `营业收入`，统一命名为 `收入` | `收入` | `收入` |
| gross_amount | 需在清洗时映射为统一 `营业额` 字段 | `营业额` | `营业额` |
| customer_paid | `顾客实付` | `顾客实付总额` | `顾客实付` |
| valid_orders | `有效订单` | `有效订单` | `有效订单` |
| avg_ticket | `实付单均价` | `单均实付` | `实付单均价` |
| exposure_users | `曝光人数` | `曝光人数` | `曝光人数` |
| visit_users | `入店人数` | `进店人数` | `入店人数` |
| visit_conversion_rate | `入店转化率` | `进店转化率` | `入店转化率` |
| order_conversion_rate | `下单转化率` | `下单转化率` | `下单转化率` |

### 5.2 标准化处理规则

- 日期统一清洗为 `yyyy/mm/dd`
- 城市统一去后缀，例如 `广州` 和 `广州市` 归一
- 百分比字符串统一转为小数
  - 例如京东 `5.53%` 转为 `0.0553`
- 空字符串数值字段统一转为 `0` 或 `NULL`
  - 汇总指标默认聚合时按 `0` 处理
  - 口径不明确字段按 `NULL` 保留

## 6. 门店映射规则基线

### 6.1 必须单独建标准门店体系

原因很直接：

- 三个平台门店名称不完全一致
- 同一门店可能存在括号写法差异
- 后续如果直接按平台门店名做统计，跨平台汇总会失真

### 6.2 最小映射模型

- `standard_store`
  - `standard_store_id`
  - `standard_store_name`
  - `province`
  - `city`
  - `district`
  - `is_online`

- `platform_store_mapping`
  - `platform`
  - `platform_store_id`
  - `platform_store_name`
  - `standard_store_id`
  - `match_status`
  - `match_method`

### 6.3 当前执行规则

- 第一阶段不做全自动智能匹配
- 先做 `半自动 + 人工确认`
- 初始映射底表直接采用：
  - `/Users/apple/窄巷口/可视化看板/Master_门店主表/门店对齐表_最新版.xlsx`
- 可先按以下规则初筛：
  - 去掉中英文括号差异
  - 统一空格、全角半角
  - 结合城市信息辅助匹配

这是更现实的做法，准确率会比强行全自动更高。

## 8. 数据库分层基线

### 8.1 采用分层，不让前端直接查原始表

- `ods_platform_daily_raw`
  - 存放各平台原始明细
  - 保留原字段和原始来源文件信息

- `dwd_platform_daily_normalized`
  - 存放字段标准化后的日级明细
  - 统一字段名、日期、百分比、小数格式

- `dim_standard_store`
  - 标准门店维表

- `bridge_platform_store_mapping`
  - 平台门店到标准门店的桥接表

- `dws_platform_daily_summary`
  - 平台日汇总

- `dws_store_daily_summary`
  - 门店日汇总

- `dws_city_daily_summary`
  - 城市日汇总

### 8.2 这样分层的原因

- 新增看板时优先新增汇总层，不改原始层
- 旧看板如果依赖固定的汇总层，受新需求影响更小
- 数据清洗和页面开发可以解耦

## 10. 风险控制基线

这一部分必须跟 `1-8` 一起定下来，否则后面会直接影响旧看板稳定性。

### 10.1 页面隔离

- 每个看板独立页面
- 每个图表独立组件

### 10.2 接口隔离

- 每个页面独立接口组
- 禁止把所有图表塞进一个大接口

### 10.3 数据隔离

- 前端只读汇总层
- 汇总层变更不能直接覆盖原始层

### 10.4 版本留档

- 每完成一个稳定模块就提交版本
- 新增模块前先备份稳定版本
